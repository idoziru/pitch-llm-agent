# pitch-wording-gamedev.md

## Назначение

Используй этот skill для вординга питчей по **разработке и изданию игр** (game studio / publisher investment-grade deck).

Это **расширение** `skills/pitch-wording.md`. Применяй его **вместе с общим skill**, а не вместо него: общий задаёт универсальные принципы вординга, этот добавляет только геймдев-специфику.

Здесь нет копий универсальных правил (заголовки-выводы, дисциплина метрик, source register, факты vs прогнозы, self-check) — они в `pitch-wording.md`.

Те же запреты, что и в общем skill: не выдумывай данные; готовый текст и данные пользователя не меняй без разрешения; пропуски помечай (`TBD` / `requires confirmation`). Этот skill не задаёт визуальный стиль и вёрстку.

---

## Расширение каркаса

Поверх универсального каркаса из `pitch-wording.md` для игрового deck добавляются разделы:

Segment economics · Current results (released titles) · Release economics · Long-tail / DLC monetization · Future pipeline · Project cards.

Нарратив История → Доказательство → Пайплайн → Предложение наполняется игровыми доказательствами: прошлые релизы доказывают повторяемость, pipeline доказывает visibility, экономика релиза доказывает модель.

---

## Геймдев-метрики и их определения

Используемые метрики: copies sold, wishlists, DAU, MAU, ARPU, first 30-day revenue, long-tail revenue, critic score, user review score.

- Для `revenue` всегда уточняй, что включено: DLC, platform fees, refunds, discounts, taxes, publisher share, региональные ценовые различия.
- Не смешивай copies sold с revenue, revenue с profit, gross с net, wishlist с продажами.
- wishlist, DAU, MAU — это proxy-сигналы спроса, не выручка. Помечай как demand signal.

---

## Current results: доказывай repeatability

Цель раздела — повторяемость коммерческого результата, а не разовый успех.

- Показывай динамику между релизами, launch performance, pre-release monetization, first 30-day revenue, long-tail revenue, return metrics.
- Reviews и critic/user score ставь рядом с commercial metrics, если они поддерживают тезис.
- Не показывай copies sold без revenue, если цель — доказать коммерческий результат.
- Не показывай revenue без cost / return / margin, если цель — доказать экономику.
- Не показывай awards без связи с коммерческим результатом.

---

## Product cards (released games)

Единый набор полей, единый порядок, единая плотность для всех released-продуктов:

product name · one-line description · release date · platforms · retail price at launch · copies sold · gross revenue · cost / return (если данные есть и слайд доказывает экономику) · critic score · user review score · awards / quotes (если релевантны) · source + as-of date.

Логотип продукта — где возможно. Неизвестное поле не выдумывай: `TBD` / `requires confirmation`.

---

## Pipeline slides

Цель — доказать visibility и repeatability, а не показать список обещаний.

Для каждого проекта: stage · release window · milestones · demand signal · главный риск · следующая точка проверки · статус достоверности данных.

Разделяй:

- confirmed projects и unannounced projects;
- base game, DLC и support period;
- стадии: announced, in production, prototype, vertical slice, concept, secured IP.

Раннюю стадию помечай явно. Estimated sales на основе internal forecast помечай как `management forecast` / `illustrative estimate`. Не выдавай pipeline за гарантированную выручку.

---

## Project cards (pipeline)

Единый набор полей:

product name · one-line description · IP / franchise · genre · current stage · release window · platforms · wishlists / demand signal · estimated sales · key milestones · main risk · next validation point · data status · source / as-of date · open issues.

---

## Operating model (геймдев)

Объясняет, почему студия способна повторять результат:

- как core team переходит от одного проекта к следующему;
- как support team поддерживает DLC и long-tail monetization;
- как centralized support снижает execution risk;
- как publishing arm усиливает модель; какие функции централизованы.

Связь с выпуском продукции, маржой, capital efficiency, снижением риска. Не подтверждено результатами — `requires evidence`.

---

## Market / segment (геймдев)

Свяжи рост конкретного сегмента / жанра / платформы с бизнес-моделью и value driver компании. Не показывай общий «игровой рынок» ради большого числа: большой рынок, который компания не может захватить, не доказывает тезис.

---

## Peer-критерии выбора

Выбирай peers по релевантным критериям и объясняй выбор: genre, platform, monetization model, IP ownership, publisher/developer model, scale, release cadence. Не подбирай peers только потому, что они удобно подтверждают тезис; спорный peer — добавь caveat или замени.

---

## Типовые игровые возражения инвестора

Поверх универсальных возражений из `pitch-wording.md` закрывай игровые:

- зависимость от одного хита / одного IP / одной команды / одной платформы;
- wishlist ≠ будущие продажи;
- gross revenue ≠ profit;
- awards ≠ commercial proof;
- announced project ≠ commercially validated;
- future release window ≠ committed revenue.

Примеры заголовков-выводов, закрывающих возражения:

- `Multi-title pipeline reduces single-release dependency`;
- `Released titles prove repeatable commercial execution`.

---

## Confidentiality (игровые проекты)

Не используй закрытые названия проектов без разрешения. Чувствительные цифры для внешнего использования заменяй диапазонами, индексами или относительными показателями. Помечай confidential данные перед финализацией.

---

## Геймдев-дополнения к финальной проверке

В дополнение к универсальному чеклисту из `pitch-wording.md`:

- нет wishlist без conversion caveat;
- нет pipeline без caveat;
- единые поля у всех product cards и у всех project cards;
- одинаковое написание product name и IP во всех слайдах;
- нет copies sold без revenue (где цель — коммерческий результат).

---

## Геймдев-ошибки для отлова

- разное или неправильное написание product name либо IP в разных местах;
- wishlist используется как proxy for sales без caveat;
- pipeline выглядит как обещание без milestone validation;
- copies sold показаны без revenue;
- gross sales и gross revenue используются как синонимы без определения;
- net revenue и gross revenue смешаны.
