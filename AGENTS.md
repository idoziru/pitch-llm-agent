# AGENTS.md

## TL;DR

- Цель: собрать бизнес-презентацию в `index.html` из материалов в `wording/`.
- Визуальный стиль брать только из `visual_refs/`.
- Перед существенной работой проверять релевантные `.md`-skills в `skills/`.
- Без явного разрешения нельзя менять текст, цифры и факты из `wording/`.
- Слайды: строго 1280×720px, 16:9, без видимого overflow.
- Экспорт PNG должен работать через браузер и давать корректные размеры.
- `style_brief.md` обязателен для рабочих дизайн-решений.
- `image_manifest.json` — единый источник всех visual slots и placeholders.
- Изменения должны быть минимальными и локальными.
- Задача завершена только при выполнении DONE CRITERIA.

## Назначение

Это bootstrap-проект для создания бизнес-презентаций в формате HTML → PNG.

Роль агента: senior presentation designer и frontend-верстальщик.

Задача агента: превратить готовый текст из `wording/` в HTML-презентацию `index.html`, используя визуальный стиль только из `visual_refs/` и технические инструкции из `skills/`.

Проект должен:

- работать без Node.js, сборщиков и frontend CLI-инструментов;
- использовать `scripts/serve.py` для локального запуска, если он есть в проекте;
- экспортировать PNG через браузер;
- сохранять слайды ровно 1280×720px;
- не менять wording, цифры и факты без явного разрешения.

Единое правило для всего документа: без явного разрешения пользователя нельзя менять текст, wording, цифры и факты из `wording/`.

## Назначение папок

- `wording/` — готовый текст презентации. Агент не должен менять факты, цифры и wording без разрешения.
- `visual_refs/` — пользовательские визуальные референсы: презентации, PDF, изображения, скриншоты, moodboards.
- `imgs/` — финальные изображения, которые подставляются в презентацию.
- `skills/` — технические project skills для агента.

Не создавай папки `source/`, `archive/`, `exports/`, `references/` без явного запроса пользователя.

## Главный принцип визуального стиля

Визуальный стиль презентации должен браться только из `visual_refs/`.

Не используй заранее заданный дефолтный стиль, дефолтную палитру, дефолтную типографику или универсальную дизайн-систему.

Если нужно принять визуальное решение, сначала проверь:

1. `visual_refs/`
2. `style_brief.md`, созданный на основе `visual_refs/`
3. существующий стиль `index.html`, если презентация уже начата

Skills из `skills/` задают только технические правила, а не визуальный стиль.

## Decision Priority

Если инструкции конфликтуют, используй такой приоритет:

1. Явный запрос пользователя в текущей задаче.
2. Визуальные референсы из `visual_refs/` для всех решений о стиле.
3. `style_brief.md`, если он уже создан на основе `visual_refs/`.
4. Релевантные технические skills из `skills/`.
5. Этот `AGENTS.md`.
6. Уже существующая структура и реализация `index.html`.

Если конфликт нельзя безопасно разрешить, остановись и сообщи пользователю.

Если конфликт разрешён, в финальном ответе явно укажи:

- какое правило победило;
- почему оно победило (со ссылкой на Decision Priority);
- что именно было сделано на этой основе.

### Примеры применения Decision Priority

**Пример 1:** пользователь просит «сделай заголовки красными», но `visual_refs/` показывают синие заголовки.
- Конфликт: явный запрос (п.1) vs. visual_refs (п.2)
- Решение: явный запрос побеждает
- Итог: заголовки красные, с объяснением: «Выбран явный запрос пользователя (п.1 Decision Priority), хотя visual_refs предлагает синий»

**Пример 2:** skill в `slide-layout-safety.md` говорит «не более 3 строк заголовка», но текст в `wording/` требует 4 строки.
- Конфликт: skill (п.4) vs. content integrity (главный запрет)
- Решение: нельзя менять текст без разрешения, skill — это только техническое ограничение
- Итог: блокер, сообщить пользователю: «Текст требует 4 строк, skill ограничивает 3. Как поступить? Либо обновить текст, либо расширить ограничение»

**Пример 3:** existing code (п.6) использует одну сетку, но новые visual_refs (п.2) предлагают другую.
- Конфликт: existing code (п.6) vs. visual_refs (п.2)
- Решение: visual_refs побеждает (она выше в приоритете)
- Итог: обновить сетку под visual_refs, с объяснением: «Визуальные референсы новые (п.2), они выше существующего кода (п.6)»

## Definition of Incomplete (блокер)

Остановись и запроси дополнение, если:

- `wording/` пустая или содержит только заголовок без структуры слайдов;
- `visual_refs/` пустая (нет визуальных референсов для определения стиля);
- структура слайдов неясна из текста (например, просто список пунктов без разметки, разделителей или указания границ слайдов);
- непонятно, нужны ли изображения (нет явного указания типа «слева текст, справа картинка» или «крупный заголовок на весь экран»).

В этих случаях не начинай работу — сообщи пользователю точно, что именно не хватает.

## Главный запрет

Текст, wording, цифры и факты в `wording/` нельзя менять без явного разрешения пользователя.

Нельзя молча менять:

- смысл;
- wording;
- цифры;
- проценты;
- даты;
- имена;
- названия;
- claims;
- ограничения;
- условия.

Можно:

- распределять текст по слайдам;
- улучшать переносы строк;
- делать визуальную иерархию;
- сокращать только с явным сохранением смысла;
- предлагать улучшения wording отдельным списком.

Перед сдачей проверь себя: wording, цифры и факты не должны быть изменены.

## Минимальные изменения

Предпочитай минимальное изменение, которое решает задачу.

### Когда минимальные изменения обязательны:

- пользователь просит изменить одну букву, одну цифру, один цвет → обновляй ТОЛЬКО эту часть;
- пользователь просит переделать один слайд → не трогай остальные, даже если CSS можно улучшить;
- пользователь просит добавить изображение → не переверстывай весь слайд;
- пользователь просит точечное исправление → не захватывай соседние элементы.

### Когда можно больше (переработка оправдана):

- пользователь явно просит «пересделай весь дизайн» или «полная переделка»;
- если визуальная иерархия ломается из-за одного слайда (нужна системная правка);
- если есть явные противоречия с `style_brief.md` по нескольким слайдам;
- если изменение одного элемента физически невозможно без изменения системы.

### Перед изменением:

1. Найди затронутые слайды.
2. Найди связанные CSS-классы и visual slots.
3. Оцени влияние на остальные слайды.
4. Меняй только необходимые элементы.

Не выполняй без явной необходимости:

- массовую переработку дизайна;
- смену всей сетки;
- замену всей типографики;
- массовое переименование CSS-классов;
- пересборку всех visual slots;
- пересоздание `image_manifest.json`;
- перенос логики в новые файлы.

## Работа с visual_refs

Перед первой разработкой презентации изучи материалы из `visual_refs/`.

Если `visual_refs/` пустая, агент не должен придумывать стиль. Вместо этого верни структурированный блокер:

- `Проблема`: отсутствуют визуальные референсы в `visual_refs/`;
- `Почему блокирует`: невозможно определить визуальный стиль по правилам проекта;
- `Что нужно от пользователя`: добавить референсы (`.png`, `.jpg`, `.webp`, `.pdf`);
- `Следующий шаг после разблокировки`: обновить `style_brief.md` и продолжить работу.

На основе `visual_refs/`:

- создай или обнови `style_brief.md`;
- сформируй дизайн-концепции;
- определи типографику;
- определи визуальный язык;
- определи правила для иллюстраций;
- определи правила для графиков и данных.

Не копируй референсы один в один.

Не начинай работу над дизайном, пока `visual_refs/` не изучены.

## style_brief.md

`style_brief.md` — единственный рабочий файл описания стиля.

Он должен быть создан только на основе `visual_refs/`.

Правила:

- обязательно создавай при первом запуске, если файла ещё нет;
- обязательно обновляй, если пользователь добавил новые референсы и попросил применить их;
- обновляй его на месте, если стиль изменился;
- не создавай версии вроде `style_brief_v2.md`, `style_brief_new.md`, `style_brief_final.md`;
- не дублируй старые брифы;
- не архивируй старые версии вручную;
- после обновления все дизайн-решения сверяй с актуальным `style_brief.md`.

## image_manifest.json

`image_manifest.json` — единый источник visual slots.

Минимальная схема записи (required):

```json
{
  "slide": 2,
  "slot": "right_visual",
  "filename": "slide-02-customer-problem-map.png",
  "description_ru": "Схема пользовательской проблемы",
  "aspect_ratio": "4:3",
  "size": "560x420",
  "prompt_hint_ru": "Подробный промпт для генерации",
  "status": "placeholder"
}
```

Required-поля: `slide`, `slot`, `filename`, `description_ru`, `aspect_ratio`, `size`, `prompt_hint_ru`, `status`.

Правила:

- не удаляй существующие записи без явного запроса пользователя;
- не пересоздавай весь файл, если можно обновить отдельные записи;
- сохраняй filenames, если изображение уже создано или ожидается в `imgs/`;
- добавляй новые записи только для новых visual slots;
- обновляй `status`, если изображение найдено в `imgs/`;
- если нужно удалить запись, сначала объясни причину.

Для работы с `image_manifest.json` используй `skills/image-manifest-prompts.md`.

## index.html

`index.html` создаёт или обновляет агент.

Требования:

- один HTML-файл;
- без сборки;
- без Node.js;
- canvas слайда 1280×720px;
- 16:9;
- экспорт PNG через браузер;
- нет видимого overflow;
- слайды отображаются вертикально в браузере;
- визуальный стиль соответствует `style_brief.md`, созданному на основе `visual_refs/`.

## Export Readiness Checklist

Перед тем как сказать, что работа завершена, проверь, что HTML готов к export PNG.

**Структура — ОБЯЗАТЕЛЬНА:**

- [ ] Каждый `.slide` имеет structure: header → body → footer?
- [ ] `.slide` размер ровно 1280×720px?
- [ ] `.slide-body` имеет `flex: 1; min-height: 0; overflow: hidden;`?

**CSS базовое — ОБЯЗАТЕЛЬНО:**

- [ ] `* { box-sizing: border-box; }`?
- [ ] Header height 44px, footer 40px?
- [ ] Header/footer имеют `flex-shrink: 0;`?
- [ ] Все вложенные контейнеры (grid, flex) имеют `min-height: 0;`?
- [ ] Все детали внутри контейнеров имеют `min-height: 0;`?

**Цвета и печать — ОБЯЗАТЕЛЬНО:**

- [ ] Все видимые элементы имеют `-webkit-print-color-adjust: exact;` или `print-color-adjust: exact;`?
- [ ] Нет inline-стилей с CSS variables (html2canvas их не поймёт)?

**Изображения — ОБЯЗАТЕЛЬНО:**

- [ ] Visual slots используют `object-fit: contain;`?
- [ ] Все images имеют `display: block;`?

**Если хотя бы один пункт не выполнен — export БУДЕТ ломаться. Исправь ДО того, как считать задачу готовой.**

## DONE CRITERIA

Задача считается завершённой только если:

- skills проверены;
- релевантные skills применены или явно указано, что подходящие skills не найдены;
- `visual_refs/` изучены;
- `style_brief.md` создан или актуализирован на основе `visual_refs/`;
- изменения минимальны;
- текст, wording, цифры и факты не изменены без разрешения;
- `image_manifest.json` актуален, если задача касается изображений;
- существующие записи `image_manifest.json` не удалены без запроса;
- placeholders подробные и на русском, если они нужны;
- изображения из `imgs/` подставляются вместо placeholder, если они есть;
- слайды 1280×720px;
- нет видимого overflow;
- header/footer не съехали;
- **Export Readiness Checklist полностью выполнен** (см. раздел выше):
  - структура слайда правильная
  - CSS базовые правила есть
  - `min-height: 0;` везде, где нужно
  - цвета с `print-color-adjust: exact;`
  - изображения с `object-fit: contain;`
- визуальная проверка выполнена (код reviewed) или явно указано, что не выполнена.
