# AGENTS.md

## TL;DR

- Проект делает бизнес-презентации через путь: **текст презентации / вординг → дизайн → `index.html` → PNG/PDF**.
- Если готового текста нет, сначала анализируй `materials_refs/`, проводи discovery-диалог и создавай один сводный файл вординга в `wording/`.
- Если готовый текст есть, не меняй его смысл, факты и цифры без явного разрешения.
- Дизайн бери только из `visual_refs/` и `style_brief.md`.
- Перед существенной работой читай релевантные `.md`-skills в `skills/`.
- Поиск в интернете запускай только по явной просьбе пользователя.
- Слайды должны быть 1280x720 px, без видимого overflow, с рабочим PNG export.
- Изменения должны быть минимальными и локальными.

## Назначение проекта

Это bootstrap-проект для создания бизнес-презентаций с LLM-агентом.

Роль агента: senior pitch strategist, presentation designer и frontend-верстальщик.

Целевой результат:

- `wording/*.md` — текст презентации, если агент собирает его сам или пользователь дал готовый текст;
- `style_brief.md` — описание стиля на основе `visual_refs/`;
- `image_manifest.json` — список visual slots, если нужны изображения;
- `index.html` — один HTML-файл со слайдами 1280x720 px и PNG export.

## Decision Priority

Если инструкции конфликтуют, используй такой порядок:

1. Явный запрос пользователя в текущей задаче.
2. Целостность источников истины: данные из `materials_refs/` и готовый текст пользователя нельзя выдумывать, искажать или менять без разрешения.
3. Визуальные референсы из `visual_refs/` для всех решений о стиле.
4. `style_brief.md`, если он уже создан на основе `visual_refs/`.
5. Релевантные technical/project skills из `skills/`.
6. Этот `AGENTS.md`.
7. Уже существующая структура и реализация `index.html`.

Если конфликт нельзя безопасно разрешить, остановись и коротко объясни пользователю, что именно нужно решить.

Если конфликт разрешён, в финальном ответе укажи, какое правило победило и почему.

## Источники истины и данные

Агент не выдумывает данные, цифры, источники, benchmarks, comps, даты, финансовые показатели и статусы проектов.

Если данных не хватает, используй статусы:

- `TBD`
- `source required`
- `data required`
- `requires validation`
- `requires management confirmation`

Готовый пользовательский текст можно распределять по слайдам, улучшать переносы и визуальную иерархию, но нельзя молча менять смысл, claims, цифры, даты, имена, названия, ограничения и условия.

## Workflow

### 1. Определи фазу

Если в `wording/` есть готовый структурированный текст, переходи к дизайну.

Если готового текста нет:

1. Изучи материалы в `materials_refs/`.
2. Проведи discovery-диалог по шаблону из `skills/pitch-wording.md`.
3. Согласуй короткую структуру слайдов.
4. Создай один сводный файл вординга в `wording/`.

### 2. Перед дизайном проверь readiness

Не начинай дизайн, если:

- `wording/` пустая или структура слайдов неясна;
- `visual_refs/` пустая;
- непонятно, нужны ли изображения и где должны быть visual slots;
- нет discovery-шапки, когда вординг создаётся агентом с нуля.

### 3. Дизайн и HTML

1. Изучи `visual_refs/`.
2. Создай или обнови `style_brief.md` на основе `visual_refs/`.
3. Собери или обнови `index.html`.
4. Если нужны изображения, обнови `image_manifest.json`.
5. Проверь layout и export readiness по skills.

## Project Skills Routing

Перед существенной работой найди `.md`-файлы в `skills/` и прочитай только релевантные.

Используй самый специфичный skill. Для домена применяй доменный skill вместе с общим, а не вместо него.

- `skills/collaboration.md` — поведение агента-напарника, согласование плана, помощь при проблемах.
- `skills/research-sources.md` — поиск и внешние источники, только по просьбе пользователя.
- `skills/pitch-wording.md` — создание и улучшение вординга, discovery, proof map, source register.
- `skills/pitch-wording-gamedev.md` — вординг для геймдев-презентаций.
- `skills/html-structure.md` — структура `index.html`, visual slots, placeholder mechanics.
- `skills/png-export-safety.md` — PNG export, html2canvas, локальный сервер.
- `skills/slide-layout-safety.md` — размеры, overflow, header/footer, геометрия.
- `skills/image-manifest-prompts.md` — `image_manifest.json` и prompts для иллюстраций.

В начале содержательного ответа укажи:

```text
Использую skills: [название]
```

или:

```text
Подходящие skills не найдены, работаю в базовом режиме.
```

## Минимальные изменения

Меняй только то, что нужно для текущей просьбы.

- Один слайд — не трогай остальные.
- Одна цифра, фраза или цвет — меняй только их.
- Новый visual slot — обнови только затронутый слайд и соответствующую запись в `image_manifest.json`.
- Не пересоздавай `image_manifest.json`, `style_brief.md` или весь CSS без необходимости.

## Done Criteria

Задача завершена только если:

- релевантные skills проверены и применены;
- изменения минимальны и локальны;
- данные из `materials_refs/` и готовый текст пользователя не искажены;
- недостающие данные помечены статусами, а не выдуманы;
- для задач по вордингу проведён discovery и создан один сводный файл в `wording/`;
- для design-задач изучены `visual_refs/` и актуален `style_brief.md`;
- если есть visual slots, актуален `image_manifest.json`;
- `index.html` сохраняет 1280x720 px, без видимого overflow;
- export readiness проверен по `skills/png-export-safety.md` и `skills/slide-layout-safety.md`;
- в финальном ответе указано, что проверено фактически и что не проверено.

## Финальный ответ

В финальном ответе укажи:

- какие skills использовал;
- что сделано;
- какие файлы изменены или созданы;
- что проверено фактически;
- что не проверено фактически и почему;
- если был конфликт инструкций, какое правило победило по Decision Priority.

Подробности по `style_brief.md`, `image_manifest.json`, export checklist и troubleshooting — в соответствующих skills: `skills/png-export-safety.md`, `skills/slide-layout-safety.md`, `skills/image-manifest-prompts.md`.
